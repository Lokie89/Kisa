
# 1. 소프트웨어 생명주기
소프트 웨어 생명 주기는 1.개발단계 2.주요활동 3.결과로 구성된다.<br>
소프트웨어 생명주기 모델 (소프트웨어 공학 패러다임) 에는 폭포수, 프로토타입, 나선형, 애자일 모델이 있다( 일반적으로 )
 
 #### 1-1. 폭포수 타입 ( 선형 )
 폭포수 모델은 한단계가 완전히 끝나야 다음 단계를 실행한다.<br>
 가장 오래된 모델이다.<br>
 각 단계별 결과물이 있어야 다음 단계를 진행할 수 있다.<br>
 
 #### 1-2. 프로토타입( 원형 )
프로토타입 모델은 실제 개발 하려하는 소프트웨어의 시제품(프로토타입)을 미리 만들어 예측하며 개발<br>
폭포수 모델의 단점( 완료 이후 나오는 오류 ) 를 보완한 모델

#### 1-3. 나선형
나선형 모델은 폭포수, 프로토타입의 장점 + 위험분석

#### 1-4. 애자일
애자일 모델은 일정한 주기를 반복하며 개발<br>
어떠한 모델에 기초가 있는 것이 아니라 서로 소통하며 일정한 주기에 반복적으로 개발하는 것을 말함<br>
설계 -> 테스트 -> 개발

 # 3. XP 기법 (eXtreme Programming)
 
 애자일 기법중 하나로 고객의 요구사항에 유연하게 대응하기 위함<br>
 요구사항 ( 사용자 스토리 ) => 계획 수립( 릴리즈 계획 ) => 반복 (개발 => 승인(테스트)) => 소규모 릴리즈<br>
 
 #### 사용자 스토리 
 고객의 요구사항
 #### 릴리즈 계획 수립 
 개발 일정
 #### 스파이크 
 요구사항 신뢰성을 높이기 위해 만드는 프로그램?
 #### 이터레이션 
 개발 일정 단위(스크럼의 스프린트)
 #### 승인 
 테스트 하여 오류 찾아내고 다음 이터레이션에 포함, 완료되면 소규모 릴리즈
 ##### 소규모릴리즈 
 소규모로 시행하면 테스트 요구사항을 실제 소프트웨어로 확인 가능 따라서 유연한 대처가 가능
 
 #### XP 실천 방법 
 Pair Programming, TDD, Whole Team, Continuous Integration, Refactoring, Small Releases
 
 # 4. 현행 시스템 파악
 #### 1. 시스템 구성 파악

 찾아낸 포인트, 키워드로 분리하여 계층형으로 작성

 시스템을 이루고 있는 하드웨어 들의 구성 파악 ( CPU, 수량, 이중화 여부 등)
 #### 6. 네트워크 구성 파악
 서버의 위치 네트워크 연결 방식 파악

 # 6. 요구사항
 #### 소프트웨어가 문제를 해결하기 위해 제공하는 서비스, 제약조건 등을 표현
전반적인 흐름, 기획의 근거, 기준, 과정의 목표 등을 알수 있음
###### 기능 요구사항
시스템이 수행해야 하는 기능
###### 비기능 요구사항
장비, 성능, 인터페이스, 데이터 ,테스트, 보안, 제약, 관리 지원, 품질 요구사항
###### 사용자 요구사항
사용자 관점 요구사항
###### 시스템 요구사항
개발자 관점 요구사항

#### 요구사항 프로세스
###### 도출
소프트웨어가 해결해야할 문제를 파악 / 개발 생명주기 동안 반복함
###### 분석
명확하지 않은 요구사항을 정리, 비용과 개발 일정을 결정, 소프트웨어의 범위 파악
###### 명세
정리된 요구사항을 기록, 기능 요구사항은 전부 빠짐없이 기록함
###### 확인
검토, 검증

 # 9. UML (Unified Modeling Language)
 시스템 개발 과정에서 개발자와 사용자 또는 개발자와 개발자 간의 의사소통을 원활히 하기위해 표준화한 객체지향 모델링 언어
 
 ### 1. 사물
다이어그램 안에서 관계가 형성될수 있는 대상
###### 구조사물
시스템의 개념적, 물리적 요소 / 클래스, 유스케이스, 컴포넌트 노드 등
###### 행동사물
시간과 공간에 따른 요소들의 행위 표현
###### 그룹 사물
요소들을 그룹으로 묶어 표현
###### 주해 사물
부가적인 설명, 제약조건

### 2. 관계
 #### 연관 관계
 2개 이상의 사물이 서로 관련 / 실선으로 표현, 방향성은 화살표로 표시, 양방향일땐 화살표만 생략
 #### 집합 관계
 하나 이상의 사물이 다른 사물에 포함, 서로 독립적 / 포함되는쪽이 포함하는쪽에 빈마름모로 연결
 #### 포함 관계
 하나 이상의 사물이 다른 사물에 포함, 서로 종속적, 생명주기 함께함 / 포함되는쪽이 포함하는쪽에 채운 마름모로 연결
 #### 일반화 관계
 하나의 사물이 더 구체적일때, 부모 자식으로 표현 / 구체적인쪽이 일반적인쪽에 빈 화살표로 연결
 #### 의존 관계
 서로 연관 관계에 있으나 서로 필요에 의해 짧게 영향을 주는 관계 / 점선 화살표로 표현 (연관관계에 점선)
 #### 실체화 관계
 사물이 할 수 있거나 해야 하는 일을 표현 / 사물이 행동에 점선 화살표로 연결 
 
 ### 3. 다이어그램
 사물과 관계를 도형을 표현, 의사소통에 도움을 줌
 #### 구조적 다이어그램
 ###### 클래스 다이어그램 : 클래스 속성, 관계를 표현, 시스템의 구조 파악 문제점 도출
 ###### 객체 다이어그램 : 인스턴스를 특정 심점의 객체와 객체 사이의 관계로 표현
 ###### 컴포넌트 다이어그램 : 컴포넌트간의 관계 인터페이스 표현, 구현 단계에서 사용
 ###### 배치 다이어그램 : 결과물, 프로세스, 컴포넌트 등의 위치를 표현, 노드와 의사소통 경로로 표현, 구현 단계에서 사용
 ###### 복합체 구조 다이어그램 : 클래스나 컴포넌트의 내부 구조를 표현
 ###### 패키지 다이어그램 : 모델 요소들을 그룹화한 패키지들의 관계 표현
 
 #### 행위 다이어그램
 ###### 유스케이스 다이어그램 : 사용자와 사용사례간의 관계 표현, 요구사항 분석때 사용
 ###### 시퀀스 다이어그램 : 상호 작용하는 시스템, 객체들이 주고받는 메세지 표현
 ###### 커뮤니케이션 다이어그램 : 시퀀스 다이어그램 + 연관 표현
 ###### 상태 다이어그램 : 객체가 클래스의 상태 변화 혹은 다른 객체와 상호 작용에 따른 상태 변화 표현
 ###### 활동 다이어그램 : 시스템의 기능 수행을 처리의 흐름 순서에 따라 표현
 ###### 상호작용 개요 다이어그램 : 상호작용 다이어그램 간의 제어 흐름 표현
 ###### 타이밍 다이어그램 : 객체 상태 변화와 시간 제약을 표현
